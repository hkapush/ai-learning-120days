## 核心笔记

- XGBoost / LightGBM 都属于 boosting 风格的树模型。
- boosting 的直觉版理解是：后一棵树专门去修正前面做错的地方，一棵一棵往后补。
- 这类模型在表格数据任务中很常见，也常常很强。
- 和前面的简单模型相比，XGBoost/LightGBM 更依赖参数设置和数据规模。
- 小数据场景下，默认参数未必能直接学起来，甚至可能根本不发生有效分裂。
- `min_child_weight` 可以理解成“对子节点分裂条件的限制强弱”。在这次极小数据实验里，把它调小有助于模型开始分裂。

## 代码/实验结果

- 初始实验结果：

```text
predictions = [0 0 0 0 0 0]
accuracy = 0.5
feature importance = [0. 0.]
```

- 现象理解：
  模型几乎没有学到有效规则，基本退化成了“全猜 0”，且没有真正利用特征做分裂。

- 调整参数后：

```python
model = XGBClassifier(
    n_estimators=50,
    max_depth=3,
    learning_rate=0.3,
    min_child_weight=0,
    eval_metric="logloss",
    random_state=42
)
```

- 调整后结果：

```text
predictions = [0 0 0 1 1 1]
accuracy = 1.0
feature importance = [1. 0.]
```

- 结果说明：
  加入 `min_child_weight=0` 后，模型终于发生了有效分裂，说明前面的主要问题不是代码错，而是默认参数对极小数据过于保守。

## 遇到的问题

| 问题 | 原因 | 解决方式 |
| --- | --- | --- |
| 模型一开始全预测成 `0` | 默认参数在极小数据集上过于保守，树没有有效分裂 | 增加树数量、调大学习率，并关键性地把 `min_child_weight` 调低 |
| 一开始容易把问题理解成“代码写错” | 输出异常但代码结构本身没问题 | 学会从“数据规模”和“模型参数”两个方向排查 |
